## Category 2: Educational (5 tweets)

### E1 — The 5 signals
> What makes a good MCP server?
>
> We score 25,000+ by 5 signals:
> • Stars (15%) — raw popularity
> • Freshness (25%) — days since last commit
> • Issue health (25%) — closed/total ratio
> • Contributors (10%) — bus factor
> • Dependents (25%) — who depends on it
>
> Freshness + dependents carry the most weight.

**[READY FOR REVIEW]** — ~278 chars

---

### E2 — Bus factor warning
> 40% of MCP servers have exactly one contributor.
>
> That means one person leaving kills the project.
>
> When we score tools, contributor count matters. A tool with 1 maintainer gets capped no matter how many stars it has.

**[READY FOR REVIEW]** — ~219 chars

---

### E3 — Stale ecosystem
> 54% of MCP repos haven't had a commit in 91-365 days.
>
> Star count doesn't tell you this. Freshness does.
>
> This is why we weight recency at 25% of the AgentRank score — the most recent commit date is the most honest signal in the index.

**[READY FOR REVIEW]** — ~244 chars

---

### E4 — Official vs community servers
> Rule of thumb for picking MCP servers:
>
> If a major vendor ships an official server (Redis, MongoDB, GitHub, Azure), use it.
>
> Official servers track their own API changes. Community forks go stale when the vendor updates.

**[READY FOR REVIEW]** — ~235 chars

### E5 — MCP vs REST decision
> When do you build an MCP server instead of a REST API?
>
> Simple rule: build MCP when your consumer is an AI agent. Build REST when it's a developer writing code.
>
> They serve different audiences. The right answer for most teams: build both.

## Category 5: Ecosystem Commentary (2 tweets)

### C1 — A2A vs MCP
> A2A and MCP are not competing protocols.
>
> MCP connects agents to tools (databases, APIs, filesystems).
> A2A connects agents to other agents.
>
> MCP is the tool layer. A2A is the coordination layer. Most production agent systems will use both.
>
> agentrank-ai.com/blog/a2a-vs-mcp-agent-protocol-comparison/
